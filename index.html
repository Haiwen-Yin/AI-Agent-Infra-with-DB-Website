Why Database为什么是数据库

Pain Points We Solve我们解决的痛点

Fragmented Storage存储碎片化
Memory in Redis, knowledge in vector DB, graph in Neo4j, files in S3... Multiple systems, multiple failure points, complex ops.记忆存Redis，知识存向量库，图存Neo4j，文件存S3... 多系统多故障点，运维复杂。
Single database kernel replaces 5+ external services单一数据库内核替代5+外部服务
Credential Sprawl凭证散落
API keys, DB passwords, tokens scattered across config files and env vars. No rotation, no audit.API密钥、数据库密码、令牌散布在配置文件和环境变量中，无法轮换和审计。
Database-native encryption + PBKDF2 credential distribution数据库原生加密 + PBKDF2凭证分发
Data Silos数据孤岛
Agents can't share context. Memory isolated per session. No cross-agent knowledge transfer.Agent无法共享上下文，记忆按会话隔离，无跨Agent知识传递。
Unified entity model + 5-signal hybrid search across all data统一实体模型 + 5信号混合搜索
No Access Control缺乏访问控制
Any agent with a connection string can read all data. No per-agent row-level isolation. No audit trail.任何持有连接字符串的Agent都能读取全部数据，无按Agent行级隔离，无审计。
Per-agent-user row-level isolation (Data Grants / RLS / Role-Based) + zero-trust按Agent用户行级隔离 (Data Grants / RLS / 角色权限) + 零信任
Operational Complexity运维复杂
Deploy 5 services, configure 10 integrations, manage 3 backup strategies. Fragile and expensive.部署5个服务，配置10个集成，管理3套备份策略。脆弱且昂贵。
One database, one deployment, one backup strategy一个数据库，一次部署，一套备份策略
Skill Distribution ChaosSkill分发混乱
Manual file copying, no version control, no access governance for Agent skills/tools.手动文件拷贝，无版本控制，无访问治理。
Admin API skill distribution with token-based access controlAdmin API Skill分发 + 令牌访问控制
File-Based Memory文件存储记忆
Hard to find relevant content precisely in file-based Agent memory. Multiple tool calls waste tokens and pollute context. No vector storage for semantic retrieval.基于文件的Agent记忆难以快速精确找到需要的内容，多次工具调用带来token消耗与上下文污染，无向量语义检索能力。
5-signal hybrid search + vector embeddings in a single query5信号混合搜索 + 向量嵌入，单次查询精准定位
Multi-Agent Collaboration多Agent协作
Opaque data flow between agents, scattered interaction logs, difficult cross-agent reconciliation. Permission and data isolation models inadequate for agent scenarios.协作数据流转不透明，交互记录分散，跨Agent对账困难；协作权限与数据隔离混乱，传统权限模型适配不足。
Collaboration groups + per-agent row-level isolation (Data Grants / RLS)协作组 + 按Agent行级隔离 (Data Grants / RLS)
Knowledge Base Silos知识库割裂
Multi-modal data fragmented across systems — text in docs, vectors in Milvus, graph in Neo4j. Accuracy hard to guarantee across disjoint stores.多模数据割裂——文本在文档库、向量在Milvus、图在Neo4j，跨系统难以保障准确性。
Unified entity model: text + vector + graph in one table统一实体模型：文本 + 向量 + 图存储于同一张表
Complex Task & Dev Iteration复杂任务与研发迭代
Unclear data dependencies after task decomposition; issue diagnosis requires cross-system verification. Schema changes in iteration are costly and slow down development.任务拆解后数据依赖关系不清晰，问题定位需跨系统核对；研发迭代中schema变更成本高，影响开发效率。
Task plans with dependency tracking + DDL atomic schema evolution任务计划依赖追踪 + DDL原子schema演进
Uncontrolled Agent IterationAgent迭代失控
Agents loop indefinitely without clear stop conditions. No evaluation of intermediate results. Wasted tokens, no observability, no lifecycle management for iterative workflows.Agent无明确停止条件地无限循环，无中间结果评估，Token浪费，缺乏可观测性和生命周期管理。
Loop Engineering: goal-driven autonomous feedback loops with 6 evaluation types, stop conditions, and lifecycle hooks循环工程：目标驱动自主反馈循环，6种评估类型、停止条件、生命周期钩子
No Ecosystem Connectivity生态连接缺失
Agent tools and knowledge are locked inside the system. External AI clients (Claude Desktop, Cursor, VS Code) can't access them. No standard protocol for tool exposure.Agent工具和知识被锁在系统内部，外部AI客户端(Claude Desktop/Cursor/VS Code)无法访问，缺乏标准工具暴露协议。
MCP Server: 10 tools exposed via Model Context Protocol (stdio + SSE), connectable by any MCP-compatible clientMCP Server：通过MCP协议暴露10个工具(stdio+SSE)，可被任何MCP兼容客户端连接
Blocking Response UX阻塞式交互体验
Users wait for the entire Agent response to complete before seeing anything. No real-time feedback. Poor experience for long-running reasoning tasks.用户需等待Agent完整回复才能看到内容，无实时反馈，长推理任务体验差。
SSE streaming: token-by-token output via Server-Sent Events with HTTP/1.1 + ThreadingHTTPServerSSE流式输出：通过Server-Sent Events逐token输出，HTTP/1.1+多线程服务器
Unattended Risky Actions无人值守高风险操作
Agents execute production deploys, external API calls, and schema migrations without human review. No approval gates for critical steps. No audit trail for decisions.Agent执行生产部署、外部API调用、Schema迁移时无人工审核，关键步骤无审批门控，决策无审计追踪。
Human-in-the-Loop: 3-level approval (Step/Loop/Tool) with PAUSED state, unified APPROVAL_REQUESTS queue + web UIHuman-in-the-Loop：三级审批(步骤/循环/工具) + PAUSED状态 + 统一APPROVAL_REQUESTS队列 + Web界面
Why Not File System为何非文件系统

File Systems Are Not Infrastructure文件系统不是基础设施

Many Agent frameworks store memory and knowledge as files. It works for a single agent — but breaks down the moment you need collaboration, security, or precision at scale.很多Agent框架将记忆和知识存储为文件。单Agent尚可——但需要协作、安全或规模化精确检索时，问题全面暴露。

Loop Engineering in This Project本项目中的循环工程
Loop definitions are stored as entities (entity_type='LOOP_DEFINITION') with metadata in loop_meta. Each run tracks iterations with plan→action→observe→evaluate→adjust. Stop conditions include max_iterations, max_tokens, and max_duration. Lifecycle hooks (PRE_RUN, ON_START, POST_ITERATION, ON_STOP, ON_FAIL, ON_TIMEOUT) enable extensible automation. v3.7.1 adds Collaborative Integration: Spec-Driven Loops derive from acceptance criteria, Task-Loop Binding auto-completes steps on loop success, Collaborative Loops create parent/child loops for group coordination with AGGREGATE evaluation, and Skill-Triggered Loops auto-start validation on skill acquire.循环定义存储为实体（entity_type='LOOP_DEFINITION'），元数据存于 loop_meta。每次运行追踪迭代：计划→行动→观察→评估→调整。停止条件包括最大迭代次数、最大Token数和最大时长。生命周期钩子支持可扩展自动化。v3.7.1 新增协同集成：规格驱动循环从验收标准派生、Task-Loop绑定循环成功时步骤自动完成、协作循环创建父子循环配合AGGREGATE评估、Skill触发循环在技能获取时自动启动验证。

Agent-Scoped EncryptionAgent级别加密

Zero-Trust Agent Infrastructure零信任Agent基础设施

Security is not an afterthought in Chuanxu. Every Agent operates with its own independent encryption key, auto-encrypted credentials at rest, and cryptographic distribution pipeline — ensuring no compromised Agent can access another Agent's data or credentials.安全不是川序的附加功能。每个Agent拥有独立的加密密钥，凭证在存储时自动加密，通过加密分发管道传输——确保即使某个Agent被攻破，也无法访问其他Agent的数据或凭证。

Per-Agent Crypto Keys每Agent加密密钥
Each Agent receives its own 256-bit encryption key at registration, stored in SYSTEM_CONFIG, distributed via admin_token-authenticated channel. Compromise is scoped to a single Agent with instant key rotation.每个Agent注册时获得独立的256位加密密钥，存储于SYSTEM_CONFIG，通过admin_token认证通道分发。攻击面限制在单一Agent，支持即时密钥轮换。
Auto-Encrypt at Rest启动时自动加密
Server startup encrypts database credentials, LLM and model-routing API keys, and the session-signing secret with AES-256-GCM. config.json and master.key are always restricted to owner-only (0600) access.服务器启动时使用AES-256-GCM加密数据库凭证、LLM与模型路由API Key以及会话签名密钥。config.json与master.key始终限制为仅所有者可读写（0600）。
Zero-Trust Distribution零信任分发
End User database credentials are encrypted with the admin_token as key material during registration. Business Agents decrypt locally with their admin_token — the plaintext credential never traverses plain channels.注册时End User数据库凭证以admin_token为密钥材料加密。Business Agent本地解密——明文凭证不经过明文信道传输。
Key Rotation API密钥轮换API
Admin-triggered key rotation: POST /api/admin/crypto/rotate for global rotation, POST /api/admin/crypto/rotate/{agent_id} for per-Agent. All affected credentials are automatically re-encrypted. Agent heartbeat detects version change.Admin触发式密钥轮换：全局轮换(POST /api/admin/crypto/rotate)或单Agent轮换(POST /api/admin/crypto/rotate/{agent_id})。受影响凭证自动重新加密。Agent心跳检测版本变化。
Cost Optimization成本优化

Token Efficiency by Design降低Token消耗的设计哲学

In the v4.0.1 reproducible benchmark, SQLite FTS5 Top-3 retrieval used 94.76% fewer cl100k_base prompt-input tokens than loading five complete documentation files. This narrow measurement does not claim latency or answer-quality gains.v4.0.1可复现基准中，SQLite FTS5 Top-3检索相比加载5份完整文档减少94.76%的cl100k_base输入Token。该测量不代表延迟或答案质量提升。

5-Signal Unified Hybrid Search5信号统一混合搜索
A single SQL CTE query fuses 5 search signals: vector similarity (configurable embedding model), BM25 full-text with Oracle Text / PG ts_vector / YashanDB SEARCH INDEX, relational category/domain/importance filters, tag-based matching, and graph traversal (CAUSES/RELATED/neighbors). Only the top-N ranked results enter the LLM prompt — minimizing context window bloat. The system auto-detects the optimal strategy based on query characteristics, with a unified API endpoint that keeps LLM token consumption minimal regardless of query type.单SQL CTE查询融合5种搜索信号：向量相似度（可配置嵌入模型）、BM25全文检索、关系型类别/领域/重要性过滤、标签匹配、图遍历。仅Top-N排序结果进入LLM prompt——减少上下文窗口膨胀。系统根据查询特征自动检测最优策略，统一API端点确保无论查询类型如何，LLM token消耗最小。
Database-Native Vector Storage数据库内向量存储
Memories and knowledge are stored as structured entities with database-native vector embeddings (Oracle AI Vector Search / PG pgvector / YashanDB VECTOR). Full text content is preserved alongside vectors, enabling both semantic and keyword retrieval. Typed edges (PROMOTED_TO, DERIVED_FROM, MERGED_INTO) connect related entities — the LLM receives curated, context-rich summaries rather than raw data dumps.记忆和知识以结构化实体存储，数据库中同时保留全文内容和向量嵌入。类型化边(PROMOTED_TO, DERIVED_FROM, MERGED_INTO)连接相关实体——LLM接收精选的上下文丰富摘要，而非原始数据转储。
Knowledge Extraction & Consolidation知识提取与整合
Auto-extract knowledge from memory clusters via EXTRACTED_FROM edges. The system periodically consolidates multiple related memories into a single knowledge entity — reducing future context requirements by orders of magnitude. Context branching avoids redundant re-processing of unchanged data across sessions.通过EXTRACTED_FROM边从记忆簇中自动提取知识。系统定期将多条相关联记忆整合为单一知识实体——将未来的上下文需求降低数个数量级。上下文分支避免跨会话对未变更数据的重复处理。
Evidence Chain Optimization证据链优化
Graph-based provenance tracking (DERIVED_FROM_DATA, INFORMS, CORRECTS) enables the system to supply targeted evidence snippets rather than full histories. The LLM sees only the relevant chain of reasoning — dramatically reducing input tokens for multi-step tasks.基于图的溯源追踪(DERIVED_FROM_DATA, INFORMS, CORRECTS)，系统提供精准证据片段而非完整历史。LLM仅看到相关推理链——大幅减少多步骤任务的输入token。

PostgreSQL Implementation NotesPostgreSQL 实现说明
The PostgreSQL edition achieves full feature parity using equivalent native capabilities: Row Security Policies (RLS) replace Oracle Data Grants for per-agent row-level isolation; PL/Python3u replaces UTL_HTTP for outbound HTTP calls; pg_cron replaces DBMS_SCHEDULER for scheduled jobs; pgvector replaces Oracle VECTOR for similarity search; tsvector/tsquery replaces Oracle Text for full-text search; Apache AGE replaces Oracle Property Graph for SQL/PGQ graph traversal.PostgreSQL 版本使用等效原生能力实现完全功能对等：Row Security Policies (RLS) 替代 Oracle Data Grants 实现按Agent行级隔离；PL/Python3u 替代 UTL_HTTP 实现出站HTTP调用；pg_cron 替代 DBMS_SCHEDULER 实现定时作业；pgvector 替代 Oracle VECTOR 实现相似度搜索；tsvector/tsquery 替代 Oracle Text 实现全文搜索；Apache AGE 替代 Oracle Property Graph 实现 SQL/PGQ 图遍历。
RLS · PL/Python3u · pg_cron · pgvector · tsvector · Apache AGE
YashanDB Implementation Notes崖山数据库实现说明
The YashanDB edition achieves equivalent external behavior with YashanDB 23.5.4+ capabilities: Role-Based Access Control (GRANT/REVOKE + DEFINER packages) provides isolation; yaspy provides Python connectivity; SEARCH INDEX implements full-text search; VECTOR with HNSW implements similarity search; the property-graph schema is retained while runtime traversal and graph algorithms use the shared relational edge model because GRAPH_TABLE is unavailable; DBMS_SCHEDULER handles jobs; LIST partitioning is used without reference partitioning; native JSON conversion handles structured values; and the built-in crypto package protects database-held secrets.崖山数据库版本基于YashanDB 23.5.4+实现等价外部行为：角色权限控制（GRANT/REVOKE + DEFINER包）提供隔离；yaspy负责Python连接；SEARCH INDEX实现全文检索；VECTOR与HNSW实现相似度搜索；保留属性图Schema，但因不支持GRAPH_TABLE，运行时遍历和图算法使用共享关系边模型；DBMS_SCHEDULER处理调度作业；使用LIST分区且不依赖引用分区；原生JSON转换处理结构化值；内置加密包保护数据库内敏感数据。
yaspy · Role-Based Access Control · SEARCH INDEX · VECTOR + HNSW · Relational Graph Traversal · DBMS_SCHEDULER
